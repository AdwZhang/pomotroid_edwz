## 目的

定义应用如何管理存储的会话历史数据，包括清除所有已记录会话的能力。

## 需求

### 需求：用户可以清除所有会话历史

系统应当提供一个 `sessions_clear` Tauri 命令，在事务中执行 `DELETE FROM sessions`，删除 `sessions` 表中的所有行。命令成功时应当返回 `Ok(())`，失败时传播字符串错误。清除后不应发出任何事件 — 该操作对运行中的计时器状态无影响。

该命令应当在关键节点使用 `[sessions]` 模块前缀发出结构化日志消息，与 `commands.rs` 中现有的日志约定一致：

- 删除前 `log::info!`，表示操作正在开始
- 删除成功后 `log::info!`，包含删除的行数
- 删除失败时 `log::error!`，包含错误详情

系统应当在 `src/lib/ipc/index.ts` 中暴露一个 `clearSessionHistory()` 包装器，调用 `sessions_clear` 命令。

#### 场景：清除会话历史删除所有行

- **当** 调用 `sessions_clear` 且 `sessions` 表包含一行或多行
- **则** 所有行应当被删除，表应当为空，一条 `log::info!` 消息应当记录删除的行数

#### 场景：空表上的清除成功

- **当** 调用 `sessions_clear` 且 `sessions` 表已为空
- **则** 命令应当返回 `Ok(())` 无错误，并记录删除了 0 行

#### 场景：失败的清除被记录为错误

- **当** 调用 `sessions_clear` 且数据库操作失败
- **则** 应当发出带有 `[sessions]` 前缀和错误详情的 `log::error!` 消息，然后将错误传播给调用者

### 需求：设置 → 系统在命名子部分下暴露数据管理操作

设置 → 系统部分应当包含一个 **数据** 子部分，由 `group-heading` 元素引入（与现有的集成、语言、日志和系统托盘子部分一致）。数据子部分应当包含两个破坏性操作：**清除会话历史** 和 **重置所有设置**。每个操作应当使用相同的两步内联确认模式：初始按钮显示后出现取消和确认控件；只有确认才触发破坏性操作；取消恢复按钮且无副作用。

清除会话历史的确认文案应当明确说明该操作是永久性的且不可撤销。

#### 场景：系统设置中可见数据子部分标题

- **当** 用户导航到设置 → 系统
- **则** 应当可见"数据"分组标题，样式与其他分组标题一致（集成、语言、日志、系统托盘）

#### 场景：清除会话历史按钮显示确认

- **当** 用户点击清除会话历史按钮
- **则** 按钮应当被替换为内联确认提示，包含取消和确认控件

#### 场景：确认清除历史并关闭确认

- **当** 用户在清除会话历史确认中点击确认
- **则** 应当调用 `clearSessionHistory()`，所有会话行应当被删除，且行应当恢复到初始按钮状态

#### 场景：取消关闭而不清除

- **当** 用户在清除会话历史确认中点击取消
- **则** 不应删除任何数据，且行应当恢复到初始按钮状态

### 需求：设置 → 关于不再包含数据管理操作

设置 → 关于部分应当仅包含静态应用元数据：版本号、相关外部链接和日志目录打开器。不应包含重置所有设置或任何其他破坏性操作。

#### 场景：关于部分没有重置按钮

- **当** 用户导航到设置 → 关于
- **则** 不应存在重置或清除按钮
